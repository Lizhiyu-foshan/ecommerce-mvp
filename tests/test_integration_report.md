## 2. 详细测试结果

### 场景1: 完整购物流程

**测试目标**: 验证从商品创建到支付完成的完整业务流程

**测试步骤**:
1. ✅ 创建商品分类
2. ✅ 创建商品(含规格)
3. ✅ 用户注册/登录
4. ✅ 添加收货地址
5. ✅ 添加商品到购物车
6. ✅ 从购物车创建订单
7. ✅ 创建支付订单
8. ✅ 支付回调处理
9. ❌ 验证订单状态
10. ❌ 验证库存扣减

**发现问题**:
- 订单状态验证失败：数据库中存储的是 `PAID` (大写)，但代码中使用的是 `paid` (小写) 进行比较。这是枚举值与字符串比较的问题。
- **库存未扣减**：订单创建后，商品库存仍然是100，没有扣减2件。

**根因分析**:
查看代码发现，`cart.py` 路由中的 `checkout_cart` 函数调用的是 `OrderService.create_order()` 方法，而不是 `create_order_from_cart()` 方法。`create_order()` 方法**没有实现库存扣减逻辑**。

```python
# routers/cart.py 第 ~320 行
order = OrderService.create_order(db, user_id=current_user.id, order_data=order_data)
```

而 `create_order_from_cart()` 方法虽然有库存扣减逻辑，但未被调用。

### 场景3: 库存扣减流程

**测试目标**: 验证并发场景下的库存扣减和超卖防护

**测试步骤**:
1. ✅ 创建商品(库存100)
2. ✅ 用户A添加50件到购物车
3. ✅ 用户B添加60件到购物车
4. ✅ 用户A创建订单并支付
5. ❌ 验证库存变为50
6. ❌ 用户B创建订单(应失败，库存不足)

**发现问题**:
- **库存未扣减**：用户A购买50件后，库存仍然是100
- **超卖问题**：用户B购买60件时，订单创建成功，但此时库存理论只剩50件

**根因分析**:
同场景1，库存扣减逻辑缺失。此外，系统缺乏并发控制机制：
1. 没有数据库级别的乐观锁/悲观锁
2. 没有分布式锁
3. 库存校验和扣减不是原子操作

**风险等级**: 🔴 **高** - 可能导致超卖，造成资金损失

### 场景5: 商品下架处理

**测试目标**: 验证商品下架后购物车的处理逻辑

**测试步骤**:
1. ✅ 创建商品并添加到购物车
2. ✅ 商品下架
3. ❌ 尝试从购物车结算(应失败)
4. ❌ 验证购物车中该商品标记为无效

**发现问题**:
- **结算未拦截**：商品已下架，但结算仍然成功
- **状态未标记**：购物车中的商品状态未正确显示为无效

**根因分析**:
1. `validate_cart_for_checkout` 函数虽然会检查商品状态，但 `checkout_cart` 路由可能绕过了验证
2. 购物车列表接口返回的商品状态字段可能未正确反映实际状态

查看代码发现 `checkout_cart` 路由确实调用了验证函数，但问题可能在于：
- 验证函数返回的是有效商品列表，但结算时使用的是原始购物车数据
- 或者验证逻辑有缺陷

**风险等级**: 🟡 **中** - 用户可能购买到已下架商品

## 4. 问题记录

### 🔴 高优先级问题

1. **库存扣减缺失**
   - 影响: 订单创建后库存不扣减，导致超卖
   - 位置: `routers/cart.py` 调用错误的订单创建方法
   - 建议: 统一使用 `create_order_from_cart()` 方法

2. **并发超卖风险**
   - 影响: 高并发场景下可能出现超卖
   - 建议: 添加数据库乐观锁或分布式锁

### 🟡 中优先级问题

3. **商品下架拦截失效**
   - 影响: 用户可能购买到已下架商品
   - 建议: 修复购物车结算验证逻辑

4. **订单状态比较问题**
   - 影响: 状态验证逻辑不严谨
   - 建议: 统一使用枚举值比较

### 🟢 低优先级问题

5. **地址创建偶发失败**
   - 影响: 影响用户体验
   - 建议: 添加重试机制和错误日志

## 5. 改进建议

### 代码修复建议

#### 1. 修复库存扣减问题

```python
# routers/cart.py - checkout_cart 函数
# 修改前:
order = OrderService.create_order(db, user_id=current_user.id, order_data=order_data)

# 修改后:
order = OrderService.create_order_from_cart(
    db, 
    user_id=current_user.id, 
    cart_items=cart_items,
    address_id=address_id
)
```

#### 2. 添加并发锁机制

```python
# services/product_service.py
@staticmethod
def deduct_stock_with_lock(db: Session, product_id: str, quantity: int) -> bool:
    """带锁的库存扣减"""
    # 使用 SELECT FOR UPDATE 获取行锁
    product = db.query(Product).filter(
        Product.id == product_id
    ).with_for_update().first()
    
    if not product or product.stock < quantity:
        return False
    
    product.stock -= quantity
    product.sales_count += quantity
    db.commit()
    return True
```

#### 3. 修复商品状态验证

```python
# routers/cart.py - checkout_cart 函数
# 在创建订单前再次验证商品状态
for item in cart_items:
    if not item.product or item.product.status != "active":
        raise HTTPException(status_code=400, detail=f"商品 {item.product.name} 已下架")
    if item.product.stock < item.quantity:
        raise HTTPException(status_code=400, detail=f"商品 {item.product.name} 库存不足")
```

### 架构优化建议

1. **事务管理**: 将订单创建、库存扣减、购物车清空放在同一个数据库事务中
2. **消息队列**: 引入消息队列处理库存扣减，提高并发性能
3. **缓存层**: 添加 Redis 缓存热门商品数据，减轻数据库压力
4. **监控告警**: 添加关键指标监控（订单成功率、库存异常等）
